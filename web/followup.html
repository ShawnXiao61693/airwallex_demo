关于这几个 Follow-up 问题的回信

感谢提问。

说明一下：这个周末我花了不少时间做了好几种不同的尝试，所以线上现在看到的不是某一个"设计好的版本"，而是一路迭代下来的结果——有些取舍是有意为之，有些是试过之后主动放弃或押后的。

在逐条回应之前，我想先讲一个贯穿始终的基础判断，本质上是个性价比的取舍：在一个周末、一个人的投入下，对一个 Revenue Enablement 的交付物来说，回报最高的地方不是把信噪比打磨到极致，而是让销售真的用起来，并把"人给反馈"这条回路跑通。

两个原因。一是岗位本身——这个位置的价值兑现在"销售用了、出了业绩"，不在"我把新闻筛得多干净"。二是更现实的一点：信噪比要想真正"被治理"，得靠真实用户的反馈来训练；而要有真实反馈，前提是得有人在用。所以系统里那些按钮（有用 / 没用、匹配客户、生成话术）不是装饰——它们既是让销售用起来的抓手，又是我日后治理信噪比唯一靠谱的数据来源。少了这一层，我对信噪比做的一切优化，都只是关起门跟 AI 自己较劲。

AE / AM 用起来 → 点按钮 · 有用/没用（埋点）
→ 反馈进库 → 反向给「信源 / 评分」打分
→ 下一份情报更准 —— 人在回路里，系统在学

顺带说一个我一开始就觉得不对劲、并因此调整了方向的假设：如果周报只是日报的聚合、月报只是把前面的新闻简编汇总，那这道题就太简单了。我不认为出题的本意是这个。所以我把日 / 周 / 月当成三种服务不同决策的产品来做，而且越往长周期走，人介入得越深、越是邀请用户参与、产生真实反馈的地方——这恰恰是这套系统价值的来源，也是上面那条判断的落点。

下面逐条回应。

问题一
你怎么评估你的降噪在变好还是变差？现在有什么方法能衡量？

我的理解：降噪不能靠我自己说"变好了"，得有可量化、可跨版本比较的方法，让质量本身变成一件可被管理的事。

现在产品里已经有一个真实信号：「有用 / 没用」反馈汇成的有用率。它能反映"我留下来的情报里，有多少真的被销售认可"。这也正是我为什么先把反馈按钮做出来——没有它，衡量降噪就只能靠感觉。

但我很清楚这个信号有个盲区：它只覆盖我留下来的，看不到我误杀的。所以要做扎实，得两条腿走：在线盯有用率（相当于精确率）；离线建一个人工标注的小样本集（golden set），每次改 prompt 或换模型就拿它跑一遍准确率 / 召回率，把"感觉变好了"变成一个能跨版本对比的数；同时定期从"被判为噪音"的那批里抽样复核，把误杀率也盯住。这样"变好还是变差"才有据可依，而不是我的一面之词。

问题二
你的 s_total 是怎么来的？能复现吗？

我的理解：这其实在问——这套系统给情报打的分，现在到底可不可信、可不可复现。

先说实话：现在的 s_total 是大模型综合相关性、时效、可行动性、来源可信度之后给的一个整体判断分，所以严格讲，它还不是一个可以稳定复现的公式。这是我要坦白的现状。

我没有一上来就把评分权重精确敲死，是有意的——那需要我比现在更懂这个行业。坦白说我还没在支付行业干过，只用过 Stripe 这类平台做过集成。在我还没有行业 ground truth 的阶段，我更愿意让 AI 先给一个能跑、方向不错的初稿，而不是假装自己能把权重定得很准。

要把它做成可信、可复现，路径是清楚的：让模型只输出几个分项，总分由代码按显式权重算（权重就摆在台面上，可调、可审计）；时效分直接用发布时间算，不让模型猜；固定模型版本、把版本号随每条分数一起记录；最后再用真实反馈把权重校准到"最能预测'有用'"。也就是从"模型给一个数"，走到"可解释、可调、可复现的分"。

问题三
如何判断一个 source 是否可信？

我的理解：信噪比的根在信源质量；这是在问我有没有真办法判断来源可不可信，而不是丢给模型瞎猜。

现在来源可信度只是模型的一个粗判，算不上真正的治理。但我想先讲一个判断：信源质量本质是投入问题，不是架构问题——你投入越多、能接的源越多，能做的跨源印证就越多，质量自然越好。Demo 阶段用单一免费源（Brave）快速验证消息流，是合理的 MVP。

真要判断可信度，我会三条一起上，而不是给单一来源打一个主观分：一是信源分层（监管官网 / 公司官方 / 一线媒体，对比聚合站 / 营销号，先手工维护几十个、再慢慢长大）；二是多源印证——同一件事被几个独立来源证实，可信度才上去，这一步顺带还把重复去掉了；三是用真实反馈反向回写信誉，长期被点"没用"或事后被证伪的来源自动降权。而且不同类型分开治理更准——官方新闻、商业新闻、客户新闻、竞品新闻、监管合规、知识点，"什么算可信、什么算信号"本就因类而异。比如竞品自己放出来的数字（"1 分钟到账"这种），默认当成它的口径、而不是事实，除非被别的来源印证。

问题四
如果今天 Brave API 挂了，日报怎么保证稳定？

我的理解：这是在问系统的可靠性——别只跑通顺利的那条路，出故障时能不能稳得住。

现在确实是单点：Brave 全挂，当天就采不到新东西，这是我还没补的。但架构上已经留好了出路。采集这层（Collector）是很薄、可插拔的——加一个源就是加一个适配器，多接两三个源，单点就没了；不管接多少源，最后都统一入同一个 Postgres，而且采集和出报是解耦的：采集定时往库里填，出报只管从库里取，采集抖动顶多让当天的池子薄一点，不会直接把报告打挂。

再加上两层兜底：人审这道闸，加上优雅降级——采集失败时不硬发一份空报告，而是顺延、复用近几天仍然有效的情报并标注清楚；当天入库量低于阈值就告警，让人在发布前就知道。我理解的"稳定"是"多源去单点 + 优雅降级 + 人在回路"，而不是赌某一个外部 API 永远不挂。

问题五
如果再给两天，先补哪三件事？

我的理解：这是在看我的优先级判断——资源有限时，我知不知道什么最该先做。

我的排序还是跟着上面那条主线走：

一、拉几个真实销售做小范围试点，把反馈闭环真正跑起来。这是这个岗位的命门，也是衡量降噪好坏的真实信号来源——没有真实用户，前面说的一切治理都是自说自话。
二、给 Postgres 做向量化。这是语义去重、检索、跨源印证的地基，这次没来得及做；信源可信度里"印证"那一步，也要从这里长出来。
三、把评分和信源信誉做成可复现、可回流。s_total 落成显式公式、固定版本可复现；再用真实反馈反向给信源和类目打分，让系统真的会自己变好。

多源兜底紧随其后。但顺序我会坚持：先能衡量、再能复现、再能闭环——衡量不了的东西，是改不动的。

以上。这套东西现在做到了"能跑、看得见效果"，但还不是"被治理的"——这是我清醒的取舍，差在哪、怎么补，我都想清楚了。很感谢这道题让我把这件事想透，期待有机会当面聊。

—— 肖亮 Shawn
